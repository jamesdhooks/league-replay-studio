# League Replay Studio — Copilot Instructions

> These instructions apply to all AI agents working on this project. Read `documentation/master-plan.md` for the full architecture and feature specification.

## Project Overview

League Replay Studio is a hybrid desktop application for producing iRacing race videos. Python 3.11 + FastAPI backend, React 19 + Vite + Tailwind CSS frontend, packaged as a native Windows app via pywebview (Edge WebView2).

## Architecture

- **Backend**: `backend/` — FastAPI async API, SQLite per-project databases (`sqlite3` stdlib), WebSocket real-time events
- **Frontend**: `frontend/` — React SPA, Vite bundler, Tailwind CSS, served by FastAPI in production
- **Desktop**: pywebview with WebView2 backend on Windows
- **Entry points**: `app.py` (GUI), `cli.py` (headless), `lrs.bat` (auto-detect)

## Critical Rules

### 1. Read the Master Plan First

Before implementing any feature, read the relevant section of `documentation/master-plan.md`. Every feature is specified there with UI mockups, database schema, API endpoints, and implementation details. Do not deviate from the plan without explicit instruction.

### 2. Follow Agent Docs

Read ALL files in `documentation/agent_docs/` before beginning work. These contain binding conventions for:
- API conventions (snake_case, Pydantic schemas)
- UI conventions (toast system, no native dialogs, dark theme, animation patterns)
- Testing guidelines (pytest, FastAPI test client, in-memory SQLite)
- Logging conventions (prefixed, essential-only, no verbose traces)
- Context system (no prop drilling, React Context for state)
- Component patterns (hooks, services, kebab-case files)

### 3. Code Conventions

**Python (Backend):**
- snake_case for all variables, functions, files, API fields
- Pydantic models for all request/response schemas
- Type hints on all function signatures
- `async def` for all route handlers
- `sqlite3` stdlib for database (NOT aiosqlite, NOT SQLAlchemy)
- Structured logging with `[Context]` prefixes
- FastAPI dependency injection for services

**JavaScript/React (Frontend):**
- Functional components only (no class components)
- React Context for shared state (no prop drilling)
- Custom hooks for reusable logic (`useProject`, `useTimeline`, etc.)
- Tailwind CSS for all styling (no CSS modules, no styled-components)
- `camelCase` for JS variables, but API responses arrive as `snake_case`
- kebab-case for filenames in `documentation/agent_docs/`

**General:**
- No `alert()`, `confirm()`, `prompt()` — use Toast/Modal system
- All animations via motion wrapper (respects `prefers-reduced-motion`)
- Dark theme by default (video editing context)
- Keyboard shortcuts for all major actions

### 4. Database

- SQLite via Python stdlib `sqlite3` — zero external dependency
- One database per project (project directory contains `.db` file)
- Global app config stored as JSON file (`config.json`)
- WAL mode enabled for concurrent reads
- Schema defined in `documentation/master-plan.md` Section 4.2

### 5. API Design

- All REST endpoints under `/api/`
- snake_case field names in all JSON responses
- Pydantic schemas for request/response validation
- WebSocket at `/ws` for real-time events (analysis progress, encoding status, pipeline updates)
- See `documentation/master-plan.md` Section 6 for complete endpoint list

### 6. Testing

- Backend: pytest + FastAPI TestClient + in-memory SQLite
- Frontend: Vitest + React Testing Library
- Test files co-located or in `test/` directories
- Mock external services (iRacing SDK, FFmpeg, OBS) at service boundaries
- See `documentation/agent_docs/testing-guidelines.md`

### 7. Git Workflow

- Conventional commits: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`
- One logical change per commit
- Branch from `main` for feature work
